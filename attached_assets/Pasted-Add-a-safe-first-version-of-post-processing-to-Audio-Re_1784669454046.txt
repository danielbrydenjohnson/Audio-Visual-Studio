Add a safe first version of post-processing to Audio Reactive Studio using Three.js built-in post-processing passes.

The app is currently working. It has:

- uploaded track mode
- live input mode
- Low / Mid / High level and hit analysis
- Hit Response controls
- geometric visual templates
- Kaleidoscope
- Kaleidoscope rotation controls
- MP4/WebM recording
- 720p, 1080p, 4K output
- landscape, portrait, square output
- live visual controls during recording

Do not rebuild the app from scratch.

GOAL

Add a proper post-processing stack that improves the visual finish of every template.

This first pass should add:

1. Bloom
2. Exposure / tone mapping handled correctly
3. A clean final output pass

Do not add trails yet.

Do not add chromatic aberration yet.

Do not add vignette yet.

Do not add grain yet.

Do not add pmndrs/postprocessing yet.

Use the built-in Three.js examples modules only, such as:

- EffectComposer
- RenderPass
- UnrealBloomPass
- ShaderPass if needed for the existing Kaleidoscope shader
- OutputPass

Inspect the installed Three.js version and use the correct import paths/API for that version.

IMPORTANT ARCHITECTURE RULES

Preserve the existing renderer architecture:

- one persistent WebGLRenderer
- one visible renderer canvas
- one scene
- one camera
- one animation loop
- one ResizeObserver
- one active template root

Do not create:

- another visible canvas
- another renderer
- another scene
- another camera
- another animation loop

The final post-processed image must render to the same visible canvas currently used by the app.

Recording must continue to capture that same canvas through captureStream().

POST-PROCESSING ORDER

Create a post-processing pipeline equivalent to:

1. Render the Three.js scene with the existing camera
2. Apply the existing Kaleidoscope effect if enabled
3. Apply Bloom
4. Apply final output / tone mapping / colour conversion exactly once

The intended order is:

RenderPass
Optional Kaleidoscope pass
UnrealBloomPass
OutputPass

If the existing Kaleidoscope implementation is currently a manual render-to-target shader, refactor it carefully into the composer pipeline or integrate it without changing the visual result.

Kaleidoscope must remain a post-process effect.

Do not rotate or transform the scene, camera, active template root, or renderer canvas to achieve Kaleidoscope.

COLOUR MANAGEMENT AND TONE MAPPING

This is important.

Do not accidentally apply tone mapping or sRGB conversion twice.

Do not accidentally skip final output conversion.

With EffectComposer, route final output through OutputPass if available.

Use ACES filmic tone mapping if supported by the installed Three.js version.

Add an Exposure control that affects the final rendered brightness in a controlled way.

Suggested default:

Exposure: 1.0

Suggested range:

0.5 to 2.0

The image should not become flat white or washed out.

Bloom should glow, but highlights should still have shape.

Add:

1. Bloom toggle
- Default: On

2. Bloom Strength
- Range: 0 to 3
- Default: 0.8
- Show value

3. Bloom Radius
- Range: 0 to 1
- Default: 0.35

- Range: 0 to 1
- Default: 0.15
- Show value

5. Exposure
- Range: 0.5 to 2.0
- Default: 1.0
- Show value

6. Reset Post Effects
- Restores only post-effect defaults

Reset Post Effects should not reset:

- template
- audio source
- uploaded file
- live input
- Low/Mid/High influence
- Hit Response
- visual density
- visual speed
- element size
- depth
- glow
- brightness
- palette
- Kaleidoscope
- recording settings

POST EFFECT SETTINGS

Add strongly typed post-effect settings.

Suggested shape:

PostEffectSettings:
- bloomEnabled: boolean
- bloomStrength: number
- bloomRadius: number
- bloomThreshold: number
- exposure: number

Keep this separate from audio settings.

It can live with visual settings if that is how the current state is structured, but keep the type clear.

LIVE CONTROLS DURING RECORDING

Post Effects are artistic controls.

These controls must remain editable during recording:

- Bloom toggle
- Bloom Strength
- Bloom Radius
- Bloom Threshold
- Exposure
- Reset Post Effects

Changing them during recording should affect the live preview and be captured in the video.

Do not restart MediaRecorder when these controls change.

RECORDING COMPATIBILITY

Do not change the MediaRecorder architecture unless absolutely required.

Do not change:

- MP4/WebM detection
- MIME type handling

- bitrate logic

- audio routing
- uploaded track recording

The post-processed final image must be what gets recorded.

PERFORMANCE

Keep the first implementation simple and safe.

Use Three.js built-in passes only.

Do not add pmndrs/postprocessing yet.

Bloom should be efficient enough for 1080p.

4K may be heavier, but should still work for short recordings.

Avoid:

- recreating composer every frame
- recreating passes every frame
- React state updates inside requestAnimationFrame
- new render targets every frame
- duplicate renderers
- duplicate canvases

Update pass uniforms/properties when controls change.

CLEANUP

On renderer cleanup / component unmount:

- dispose composer resources where appropriate
- dispose render targets
- dispose custom shader materials if Kaleidoscope is refactored into a pass
- preserve existing template cleanup
- preserve existing renderer cleanup
- preserve recording cleanup
- no WebGL memory leaks
